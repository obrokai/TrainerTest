# 遠距互動健身教練系統 — 執行計畫

本文件提供開發團隊在未來 6 週內的執行步驟、交付物與工具建議，協助從概念快速落地成為可驗證的 MVP。

## 0. 團隊準備
- 指派角色：前端工程師、後端工程師、AI/電腦視覺工程師、產品/設計、DevOps。
- 建立專案管理工具（如 Jira、Linear 或 Notion），定義使用者故事與工作項目。
- 建立共同開發基礎：Git 版控、CI/CD（GitHub Actions）、共用設計系統。

## 1. 第 1 週 — 需求確認與原型設計
- 與教練與學生訪談，整理高優先需求與痛點。
- 使用 Figma 建立互動式原型，包含教練端標註工具、學生端提示。
- 明確 MVP 成功指標（如延遲 < 300 ms、姿勢偵測準確度 > 80%）。
- 產出：需求文件、線框圖、技術風險清單。

## 2. 第 2 週 — 基礎架構與串流骨架
- 搭建訊號伺服器（可先用 LiveKit Cloud 或自建 Janus 範例）。
- 建立前端專案骨架（React + TypeScript）與後端 API（FastAPI/Node.js）。
- 完成雙向視訊串流 PoC，確保可在內網環境運作。
- 產出：可連線的視訊 Demo、部署腳本、初版技術文件。

## 3. 第 3 週 — 標註與同步功能
- 實作教練端 Canvas 標註（線條、箭頭、角度扇形）。
- 使用 WebSocket 或 WebRTC DataChannel 將標註資料同步至學生端繪製。
- 建立即時聊天室或語音提示功能。
- 產出：可同步標註的互動 Demo、使用者故事驗收測試。

## 4. 第 4 週 — 姿勢偵測與回饋
- 導入 MediaPipe Pose 或 MoveNet，在學生端取得骨架資訊。
- 設計規則引擎計算關節角度並提供簡單提示（如角度區間提醒）。
- 與教練共同調整提示語氣與呈現方式，避免資訊過載。
- 產出：姿勢偵測與提示 Demo、模型效能報告。

## 5. 第 5 週 — 整合與穩定度測試
- 串接錄影保存（可暫時存於雲端 S3 測試桶）。
- 進行整合測試（E2E 流程、延遲、訊號穩定度）。
- 建立自動化測試：前端單元測試、後端 API 測試、CI 流程。
- 產出：整合測試報告、優先修正清單。

## 6. 第 6 週 — 用戶試用與迭代
- 招募 3-5 組教練/學生實際試課。
- 收集回饋並分析使用資料（標註頻率、姿勢提示觸發次數）。
- 根據回饋調整體驗、排定下一輪開發（手勢控制、課程管理等）。
- 產出：使用者回饋報告、下一版產品路線圖。

## 附錄：快速啟動指引
- **開發環境**：Node.js 18、Python 3.10、Docker（本地測試時使用）。
- **推薦工具**：
  - 版本控制：Git + Conventional Commits。
  - 測試：Vitest/Playwright（前端）、Pytest（後端）。
  - 監控：Sentry（前端/後端）、Grafana（串流品質監控）。
- **風險緩解**：
  - 早期即測量網路延遲與封包遺失率。
  - 與法務確認錄影存取權限與個資保存政策。
  - 預留替代方案（如改用第三方串流服務）。

依上述時程執行，可在 6 週內完成具備視訊、標註與基礎姿勢回饋的 MVP，後續再逐步擴充高階功能。
